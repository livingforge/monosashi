# monosashi（ものさし）

**AIエージェント / スキル / ハーネス / 知識ドキュメントの静的評価フレームワーク**

monosashi は、AI成果物一式（コード・設定・プロンプト・ドキュメント・テスト・eval定義）を **実行せずに読み解き**、生成AIを判定者（LLM-as-judge）としてトラック別ルーブリック（0〜4）で採点するフレームワークである。評価対象は実行時性能ではなく、**成果物に残された設計・文書の証拠から読み取れる作り込みの度合い** である。

- **非実行評価** — コミット済みの成果物のみを対象とする。判定エージェントには実行および別エージェント起動の権限を付与せず、非実行性を構造的に担保する。
- **単一ラベルへ集約しない** — 「主種別＋副成分」のような単一の呼称にまとめず、能力軸ごとに独立して判定し、該当するトラックをすべて適用する。
- **宣言と実体の乖離（M2）を自動算出** — 宣言種別（`declaredType`）と実能力（`axes`）から機械的に導出する。平均には含めず、常に独立した指標として併記する。

> 実行時の成功率・レイテンシ・コスト・実カバレッジは評価対象外とする（別の仕組みで補完する）。コミット済みのカバレッジレポートや eval 結果は静的に参照可能であり補助的証拠として利用できるが、主たる証拠は常にテスト・eval の設計品質である。

## 評価軸（トラック）と採点スケール

能力プロファイルに基づき、該当するトラックの項目のみを採点する（全23項目）。

| トラック | 内容 | 適用条件 |
|---|---|---|
| **M** メタ / 種別整合 | 自己識別と境界の明示、宣言と実体の整合（M2） | 常時 |
| **G** 共通 | 契約・エラー処理・ガバナンス・ドキュメント・再現性 | 常時（コード経路が無ければ一部 N/A） |
| **A** エージェント特性 | 制御設計・ガードレール・権限管理・人間の介在・観測性 | 自律制御が有/部分 |
| **S** スキル特性 | カプセル化・テスト設計・eval定義 | カプセル化が有/部分 |
| **H** ハーネス特性 | 評価カバレッジ基盤・多層防御・観測基盤 | 横断層が有/部分 |
| **K** 知識 / ドキュメント特性 | 追従性・網羅率・例の正しさ・鮮度・機械可読性 | 知識が主または相当量 |

**採点スケール（全項目共通）**

| Lv | 名称 | 意味 |
|----|------|------|
| 0 | Absent | 痕跡なし／該当する成果物が存在しない |
| 1 | Initial | 散発的・場当たり的で再現性がない |
| 2 | Defined | 主要部分が明示的に定義・実装されている |
| 3 | Managed | 観点が分離・体系化され、意図を追跡できる |
| 4 | Optimized | 失敗知見の反映に加え、観点固有の検証・強制が成果物として存在し、裏付けがある |

- **N/A**: その種別では当該項目が原理的に存在しない場合に用いる。集計から除外し、欠落としての減点は行わない。
- 各項目の詳細なレベル定義は [framework.md](.claude/skills/monosashi-eval/docs/framework.md) を参照。

## リポジトリ構成

| パス | 内容 |
|---|---|
| `.claude/` | Claude Code 向け配布物。判定エージェント群（`agents/monosashi/`）、`monosashi-eval` スキル（`skills/monosashi-eval/`。ツール一式・ルーブリック・docs）、パックカタログ（`package-meta/monosashi/`。運用・管理規約）。 |
| `.github/` | 上記と同一内容のミラー配布。 |
| `eval/` | 評価の適用例。monosashi を実エージェント（経費精算）および monosashi 自身に適用した **スコアカード** と、静的評価のフィードバックが実行時品質に与える影響を測定した **実証レポート**（[eval/dynamic/REPORT.md](eval/dynamic/REPORT.md)）。 |

`.claude/` と `.github/` はビルドで生成される配布物である（元の `src/*.ts` および `skill-src/` は本リポジトリに含まない）。配布物は `node scripts/*.mjs` でそのまま動作し、追加インストールを要しない（TOON コーデックは `scripts/toon-vendor.mjs` に同梱）。

## 使い方

カスタムエージェント `monosashi-conductor` を起動し、評価対象のパスを指定して依頼する。

```bash
claude --agent monosashi-conductor
```

起動後、評価対象フォルダのパスを指定する。

```
<評価対象のフォルダパス（各自で記載）> を評価して
```

以降、エージェントがインベントリ → 読解 → トラック適用 → 採点 → 集計の多段処理を自動で進める。最終的に、トラック・領域別レーダーチャート、M2 乖離フラグ、主要指標、項目ごとの根拠と証拠を収めた自己完結型 HTML レポート（`scoreboard.html`）のパスと一行要約を返す。出力先や判定役の呼び出しはすべてエージェント内部で処理されるため、コマンドを個別に構成する必要はない。

> メイン会話から直接実行する場合は、`monosashi-eval` スキルを呼び出すことで同一処理を実行できる（判定役は深さ1のサブエージェントとして起動する）。ただし `monosashi-conductor` 自体を *サブエージェントとして* 呼び出してはならない。Claude Code ではサブエージェントがさらにサブエージェントを起動できず、判定処理が動作しないまま静かに失敗するためである。

### 評価プロセス

エージェントが内部で進める各ステージは以下のとおりである（各ステージを個別に実行する必要はない）。

| 段 | 内容 | 担い手 |
|---|---|---|
| S0 | インベントリ（成果物と宣言ラベルの収集、全項目リストの生成） | ツール |
| S1 | **一度のみ読解** し、能力プロファイルと全項目の証拠パックを作成 | 判定役（surveyor） |
| S2 | トラック適用、ルーブリック本文の埋め込み、**M2 の機械的算出** | ツール |
| S3 | 項目別採点（根拠の引用と確信度つき）。全項目をパスA、必要箇所のみパスB/C | 判定役（judge） |
| S4 | 集計（トラック・領域別レーダー、用途別重みづけ、M2 フラグ、突合） | ツール |
| S4.5 | レポート出力（`scoreboard.html`） | ツール |

**S1 は必須の関門であり、これを経ずに S3（採点）へは進まない。** 成果物の読解は S1 の一度のみで、以降のステージは再読しない（採点はプランに埋め込まれたレベル定義と証拠パックのみに基づく）。

### 採点ルール（厳守事項）

- **根拠の引用は必須とする。** 引用できない採点は `confidence: "low"` とし、原則としてスコアを上げない。
- レベル定義の本文を **そのまま** プロンプトに含め、温度は低く固定する。
- 対象に該当能力軸が存在しない種別は **N/A** とする。
- 領域・トラックごとの平均を **レーダーチャート** で示し、単一スコアに集約しない。**M2 は常に独立指標** として併記する。
