# 静的評価フィードバックは非構造化入力下の LLM エージェント実行品質をどう変えるか — 推論型 vs スクリプト型 × モデル能力、および思考予算レバーの比較

*本レポートは `eval-run/runs/` の生データから `make-report.mjs` により自動生成（集計対象 計 150 rep）。*

## 概要

業務向け LLM エージェントの実行評価には正解データ・グレーダ・多数回の反復実行が必要であり、開発ループの早い段階で品質リスクを特定できる手段が求められる。本研究は、成果物バンドルを**実行せずに**読解・採点する静的評価フレームワーク monosashi のフィードバックが、エージェントの実行品質 — 精度・出力安定性・コスト・レイテンシ — をどれだけ改善するかを、**実務に即した非構造化テキスト入力**（経費申請メール・請求書文書）の条件で定量化する。判定ルールを自然言語でプロンプトに記述し、LLM が推論と（必要に応じた）自前のその場限りスクリプトで判定する**推論型**（改善前 v1。正規のルール正本・スキーマ・テストを持たず、判定の手順・道具立てが実行ごとに変わる）に monosashi 静的評価を適用し、その指摘（判定の再現性保証なし・テストなし・入力境界検証なし・ルール正本の不在）を反映して、判定を決定的スクリプトに外部化した**スクリプト型**（改善後 v2 を用いる v3＝テキストを契約 JSON へ構造化 → スクリプトで判定）を作成した。両方式をモデル能力 2 水準（Haiku 4.5 = 弱、Opus 4.8 = 強）と掛け合わせた 2×2 要因計画（C1〜C4）に、推論型/opus の思考予算を上げた追加セル（C2H）を加えた全 5 セル・各 N=30 回（計 150 rep）で実行評価した。

結果、静的評価フィードバック反映後のスクリプト型は (1) **精度**: 弱モデルで acc 2.067/4 → 4.000/4（Cohen's d = 5.25）と崩壊が解消し、(2) **安定性**: 全ケース distinct=1・標準偏差 0.000 の完全な決定性を両モデルで獲得し、(3) **コスト**: 満点群の中で最安（$0.0641/rep）に到達した。対する推論型は弱モデルで崩壊（C1 acc 2.067/4）、強モデルでも acc 3.933/4 と高いが出力が揺れる（expense-mixed で distinct 5）。推論型の脆弱性は思考予算を上げれば救えるが（C2H: 思考予算を上げると acc 3.933 → 4.000・全ケース distinct=1 に到達、d = 0.26）、その代償は全セル中最大のコスト（$0.2985/rep）であり、**同じ満点・同じ完全決定性を、スクリプト型は最も弱く最も安いモデルで達成する**（C3 $0.0641/rep）。静的スコアの改善（overallMean 1.65 → 3.05 /4）は、実行評価で推論型が実際に崩れた箇所（決定性・算術・ルール適用）を実行前に正しく予告していた。すなわち、推論型の確率的な脆弱性に対しては「計算資源（思考予算）で殴る」レバーも有効だが、能力の弱いモデルでも崩れない構造的な解は「判定をコミット済みの決定的スクリプトに委ねる」設計側にあり、静的評価はその設計上の弱点を実行コストゼロで特定できる品質ゲートとして機能する。

## 1. 背景と目的

LLM エージェントの品質を実行評価（多数回実行して出力を採点する方式）だけで担保しようとすると、(1) 正解データとグレーダの整備、(2) 統計的に意味のある反復回数（本研究では計 150 rep、参考換算 $28.69）、(3) 実行ハーネスの構築が必要になり、開発ループの後段でしか品質リスクが見えない。一方、静的評価 — 成果物（コード・設定・プロンプト・ドキュメント・テスト）を実行せずに読解・採点する方式 — は、これらをすべて省略して開発の早い段階でフィードバックを返せるが、その指摘が実行時の品質改善に実際につながるのかは自明ではない。

本実験は、静的評価フレームワーク monosashi のフィードバックを介した改善前後のエージェント（推論型 → スクリプト型）を、モデル能力 2 水準と掛け合わせた 2×2 要因計画で**非構造化テキスト入力**のもとに実行評価し、さらに推論型の脆弱性が「計算資源を増やす」だけで救えるかを試す追加セルを設けて、次の研究課題を扱う。

- **RQ1（精度・安定性への効果）**: 静的評価フィードバックを反映した改善（判定のスクリプト外部化）は、実行時の精度と出力決定性をどれだけ改善するか。その効果はモデル能力によってどう変わるか。
- **RQ2（コスト・速度への効果）**: 同改善はツール呼び出し・トークン・金額コスト・レイテンシにどのような影響（改善とトレードオフ）をもたらすか。
- **RQ3（静的評価の予測妥当性 — なぜ静的評価か）**: 静的スコアの改善（推論型 → スクリプト型）は実行評価での改善を予測したか。実行評価に対して静的評価が持つ固有の価値は何か。
- **RQ4（思考予算レバー vs 構造的解）**: 推論型の確率的な脆弱性は、判定方式を変えずに思考予算（thinking budget）を上げるだけで救えるか（C2H）。救えるとして、そのコストは構造的解（スクリプト化）と比べてどうか。

## 2. 対象システム

### 2.1 タスク定義

経費精算ドメインの 2 種類の判定タスクを対象とする。

1. **経費明細の承認判定**: 各明細（item）をルール R1〜R6（期間外 / 不正金額 / 領収書必須 / 不明カテゴリ / 交際費上限 / 重複疑い）で評価し、status（approved / rejected / needs_review）と reason コード列、および件数・承認合計額（approvedAmount）の summary を JSON で出力する。
2. **請求書のインボイス検証**: 登録番号形式・税額計算（税率グループごとの切り捨て）・支払期日・税率の 4 検査（V1〜V4）を行い、valid と error コード列を JSON で出力する。

全セルとも入力は構造化済み JSON ではなく**非構造化テキスト**（経費申請メール・請求書文書）で与える。実務では判定の前段に必ず「非構造データの構造化」が存在するため、その現実的な前段込みで両方式を比較する。

### 2.2 推論型（改善前 v1）: LLM が推論・自前ツールで判定

判定ルールの全文を Claude Code の Skill（`expense-agent-v1/.claude/skills/expense-audit/SKILL.md`）に自然言語で記述し、LLM が非構造化テキストを読んで判定する。テキストの読み取り（転記・正規化）・ルール適用・算術・整形のすべてを LLM が担い、その過程で LLM は必要と判断すれば自前のその場限りスクリプト（Bash で python/node を実行する電卓・集計コード等）も随時用いる — 実務でツールを与えられたエージェントに即した現実的な挙動である。ただし判定の正本となるルールコード・入出力スキーマ・テストは持たず、ルールは SKILL.md の自然言語記述のみに存在し、判定の手順・道具立ては実行ごとに LLM が即興で決めるため再現性が保証されない。静的評価フィードバックを受ける前の素朴な実装である。

### 2.3 スクリプト型（改善後 v3）: 構造化 → 決定的スクリプト判定

推論型を monosashi で静的評価した結果（§3）をフィードバックして改善した版。LLM の役割を「非構造化テキストを入力契約 JSON へ忠実に構造化し（転記と表記の機械的正規化のみ。値の補正・推測・判定は禁止。誤りに見える値も修正しない）、判定スクリプト `audit.mjs` を実行して stdout の JSON を改変せずパススルーする」ことに限定する。閾値・コード体系は `config/rules.json` を正本とし、入出力は JSON Schema による境界検証、ユニットテスト・回帰評価（eval）を同梱する（`expense-agent-v2` バンドル）。判定そのものはコミット済みの決定的スクリプトに固定されるため、推論型との差は「コードを使うか否か」ではなく（推論型も随時その場限りのコードを使う）、**判定を担うのが実行ごとに即興で変わる LLM 主導の手続きか、版管理された正本・スキーマ・テストを備えた決定的スクリプトか**に帰着する。

| 構成要素 | 推論型（改善前 v1） | スクリプト型（改善後 v3） |
|---|---|---|
| 判定主体 | LLM（推論＋随時その場限りのスクリプト） | `audit.mjs`（版管理された決定的スクリプト） |
| ルール正本 | SKILL.md の自然言語記述 | `config/rules.json`＋スキーマ |
| 入力境界検証 | なし（プロンプト指示のみ） | JSON Schema＋`input_error` 契約 |
| テスト | なし | ユニットテスト＋回帰 eval |
| LLM の役割 | 解釈・計算・整形のすべて | テキストの構造化＋スクリプト実行＋パススルー |

### 2.4 思考予算レバー（C2H）

推論型/opus（C2）の唯一の差分として、サブエージェントの思考予算（thinking budget）を既定（low）から high（16,000 tokens）へ上げたセルを C2H とする。判定方式・ペルソナ・モデル・入力はすべて C2 と同一で、変動要因は思考予算のみ。「推論型の脆弱性を、設計を変えずに計算資源だけで救えるか」を測る対照である（RQ4）。

## 3. 介入: monosashi 静的評価とフィードバック

開発は「設計 → 受け入れテスト作成 → 推論型実装 → **monosashi 静的評価 → フィードバック** → スクリプト型実装」の順で行い、設計・実装・静的評価はすべて Claude Opus 4.8 を使用した。本研究の「介入」はこの静的評価フィードバックであり、推論型 → スクリプト型の差分はすべて静的評価の指摘に対応する改善として実装された。

monosashi は、エージェント／スキルの成果物バンドル（コード・設定・プロンプト・ドキュメント・テスト）を実行せずに読解し、トラック別ルーブリック（0〜4）で採点する静的評価フレームワークである。推論型（`expense-agent-v1`）／スクリプト型（`expense-agent-v2`）のバンドルのスコアは次のとおり（`eval-out/*/scorecard.toon` から自動抽出）。

| 指標 | 推論型 v1 | スクリプト型 v2 |
|---|---|---|
| overallMean（全基準平均 /4） | 1.65 | 3.05 |
| weightedIndex | 1.67 | 3.1 |
| M（メタ／宣言整合） | 2 | 4 |
| G（ガバナンス／文書） | 1.5 | 3.2 |
| A（アーキテクチャ） | 1.75 | 3 |
| S（安全性） | 1 | 3.33 |
| H（ハーネス／テスト） | — | 2.33 |
| K（知識／ドキュメント） | 2 | 3 |

静的評価の主要な指摘と、それに対応するスクリプト型の改善・本実行評価での検証対象は次のとおり。

| 静的評価の指摘（推論型） | 対応するスクリプト型の改善 | 実行評価での検証 |
|---|---|---|
| 判定の再現性保証なし（毎回 LLM が推論） | 判定を決定的スクリプトに外部化 | 出力決定性（distinct・σ、§5.2–5.3） |
| 算術・集計を LLM に依存 | 合算・検算をスクリプトで実行 | approvedAmount の算術精度（§5.4） |
| ルール正本の不在（自然言語記述のみ） | `config/rules.json` を正本化 | ルール適用の一貫性（§5.3 請求書ケース） |
| テストなし | ユニットテスト＋回帰 eval 同梱 | —（回帰防止の仕組み。本実行評価の直接対象外） |
| 入力境界検証なし | JSON Schema＋`input_error` 契約 | —（不正入力は本実行評価の入力に含まれない） |

本実行評価は、このフィードバックが実挙動（精度・安定性・コスト・速度）の改善として現れたかを検証する位置づけにある。

## 4. 実験方法（実行評価）

### 4.1 要因計画

2×2 の 4 セル（C1〜C4）に、思考予算レバーの追加セル（C2H）を加えた計 5 セル。各セル N=30 回の独立実行（計 150 rep）。正解・グレーダ・難易度・温度・入力（全セル非構造化テキスト）は固定し、変動要因は「静的評価フィードバックの前後（推論型 / スクリプト型）」「モデル（Haiku 4.5 / Opus 4.8）」、および C2H では「思考予算（low / high）」のみ。モデル軸を設けるのは、静的評価フィードバックの効果量がモデル能力に依存するか（弱モデルほど効くのか）を測るためである。

| セル | 方式 | モデル | 思考予算 | 判定経路 |
|---|---|---|---|---|
| C1 | 推論型（改善前） | Haiku 4.5（弱） | 既定 | LLM が推論（＋随時その場限りスクリプト）で判定 |
| C2 | 推論型（改善前） | Opus 4.8（強） | 既定 | 同上 |
| C2H | 推論型（改善前） | Opus 4.8（強） | **high** | 同上（C2 と思考予算のみ相違） |
| C3 | スクリプト型（改善後） | Haiku 4.5（弱） | 既定 | LLM が構造化 → `audit.mjs` 実行＋パススルー |
| C4 | スクリプト型（改善後） | Opus 4.8（強） | 既定 | 同上 |

### 4.2 テストケース

基本テストでは天井効果が出るため、算術・規模・ルール網羅を強化した難問 4 ケースを使用した。各ケースは同一の値を自由テキスト（申請メール風・請求書文書風）で記載した `inputs/<id>.txt` として与える。

| id | 種別 | 規模 | 主に試す能力 |
|---|---|---|---|
| test-02 | 経費 | 28 明細 | 全ルール混在の分類＋複数 reason 順序＋件数集計＋合算＋重複検出 |
| test-04 | 経費 | 20 明細 | 20 個の整数の純合算（approvedAmount）。算術の素の強度 |
| test-01 | 請求書 | 14 行 | 税率グループごとの切り捨て検算 |
| test-03 | 請求書 | 10 行 | V1〜V4 全該当の検出と順序 |

テキストはカンマ区切り金額（1,000円）・%表記税率（10%→0.1）・「領収書あり/なし」・全件一括注記（「領収書は全件あり」）・年省略を含む複数形式の日付（2026/5/1、5月1日）など、機械的な正規化を要する表記ゆれを含み、文中に明細行が混在する。誤りを含む値（INV-H-B の税額＝税抜額、支払期日＜発行日など）もテキストに忠実に記載されており、推論型が「善意で修正」したりスクリプト型の抽出が値を直したりすれば採点で検出される。全セルで値の差は一切ないため、**gold（`ref/`）と採点系は全セル共通**である。

### 4.3 ブラインド規律

被評価エージェントに渡すのは入力（`inputs/` のテキスト）と自身の SKILL.md（スクリプト型はスキーマ・自分が書いた tmp も）だけ。構造化版 JSON（`inputs/`）・正解（`ref/`）・グレーダ（`grade.mjs`）・メタ情報は評価資産として隔離し、エージェントのプロンプトに含めず読ませない。

### 4.4 実行ハーネス

各 rep は Claude Code のサブエージェントとして、配備版エージェントを忠実に再現した persona プロンプト（推論型: ルール全文に従いテキストを読んで判定、スクリプト型: テキスト構造化〔値の補正・判定禁止〕＋スクリプト実行＋パススルー）を指定モデルで起動して実行した。C2H のみ思考予算を high に指定。各ケースの判定結果 JSON はサブエージェントが per-case ファイル（`tmp/<cell>-rep<rep>-<id>.result.json`）へ Write し、最終メッセージに封筒で再掲しない（弱モデルで外側 JSON 封筒の崩れが正答を 0/4 に沈める失敗モードの除去）。実行はバックグラウンド Workflow によるバッチで行い、`collect.mjs` が全サブエージェントのトランスクリプトから結果 JSON とテレメトリを抽出して `runs/*.json` に集約する（最終収集 150/150 rep。実行中に利用量制限等で結果未出力のまま終了した rep は採点対象に入らず、同番号で再実行して回収した — 同一 rep 番号が複数ある場合 `collect.mjs` は結果 JSON を持つ最新のトランスクリプトを採用する）。

### 4.5 採点指標

gold はスクリプト型の `audit.mjs` 出力（`ref/`）。採点は判断そのものの正しさを測る **acc（実質正答）** で行う。

- **acc（実質正答）**: 経費＝summary（件数＋approvedAmount）と各 item の status が一致。請求書＝valid と error コードの**集合**（順不同）が一致。
- **fieldRecall**: gold の全リーフフィールドのうち一致した割合（部分点）。
- **distinct**: 同一ケースの全 rep 出力の異なり数（1 = 完全決定的）。

reason/error 配列の列挙順序・複数コード併記の網羅といった「gold 実装の出力規約の再現」は採点対象に含めない。gold がスクリプト型の `audit.mjs` 出力である以上、この規約はスクリプト型の実装挙動そのものであり、規約一致まで要求する採点は推論型のプロンプト記述からは原理的に到達できず構造的に不公平となるためである（§7）。

### 4.6 コスト指標

トランスクリプト実測から rep ごとに、ツール呼び出し数（tool_uses）、レイテンシ（最初と最後のレコードの時刻差）、トークン内訳（input / cacheWrite / cacheRead / output。1 ターンが複数レコードに重複保存される usage を message.id でユニーク化して合算）を取得。USD は参考単価（/MTok: Opus in $5 / out $25 / cacheW $6.25 / cacheR $0.50、Haiku in $1 / out $5 / cacheW $1.25 / cacheR $0.10）で換算した参考値である。

### 4.7 統計処理

標準偏差は母集団標準偏差（分母 N。実行した全 rep が母集団）。ケース別正答率には Wilson スコア法による 95% 信頼区間を付す。セル間の per-rep acc 差の効果量は Cohen's d（プールド標準偏差）で示す。

## 5. 結果

### 5.1 主要結果: 推論型 vs スクリプト型の精度

**acc（実質正答 /4、N=30 平均）**

| | Haiku 4.5（弱） | Opus 4.8（強） |
|---|---|---|
| **推論型**（LLM が推論・自前ツールで判定） | C1: **2.067** | C2: **3.933** |
| **スクリプト型**（構造化→スクリプト判定） | C3: **4.000** | C4: **4.000** |
| 推論型 + 思考予算 high | — | C2H: **4.000** |

## 6. 考察

### 6.1 RQ1: 静的評価フィードバックの精度・安定性への効果

静的評価フィードバックを反映したスクリプト型は、acc で両モデル 4.000/4・σ 0.000・全ケース distinct=1 を達成した。改善幅はモデル能力に強く依存し、弱モデルでは推論型 2.067 → スクリプト型 4.000（d = 5.25）の崩壊回復、強モデルでは 3.933 → 4.000（d = 0.26）の僅差にとどまる。これは静的評価フィードバックの価値が「いま使っているモデルでの平均精度向上」ではなく、**モデル能力への依存そのものの除去**にあることを意味する。強モデルで開発・検収している間は推論型の弱点はほぼ見えず（C2 acc 3.933/4）、モデルのダウングレードや更新で初めて顕在化する。静的評価はこの潜在リスクを、実行では観測しにくい段階で指摘していた。懸念された反証ポイント「弱モデルではスクリプト型のラッパー（構造化・スクリプト呼び出し・パススルー）自体が崩れる」は観測されず、弱モデルはツール操作の手際こそ揺れるが、最終出力は全 rep で不変だった。

### 6.2 改善前（推論型）の失敗構造 — 静的評価の指摘との対応

推論型の劣化は単一の現象ではなく、静的評価の指摘に対応する 2 層で生じた。(1) **算術層の崩壊**（指摘:「算術・集計を LLM に依存」）: expense-sum で C1 は誤り 22/30・誤値 16 種と、多数整数の合算が確率的に全滅する。(2) **ルール適用層の不安定**（指摘:「ルール正本の不在」「再現性保証なし」）: 算術負荷の小さい請求書 2 ケースでも、C1 は invoice-all-errors で acc 23/30、invoice-rounding で 29/30 と判定自体が揺れる。強モデル（C2）では (1) はほぼ消え（sum 29/30）、(2) も大きく改善するが、出力決定性は残らない（expense-mixed distinct 5）。推論型（LLM 主導の判定）の品質は、自前ツールの利用を含めてもモデル能力に対して連続的に劣化し、どの水準でも確率的成功の域を出ない — 静的評価はこの構造を、出力を 1 つも見ることなく成果物の設計から指摘していた。

### 6.3 RQ2: コスト・速度への効果

静的評価フィードバック反映はコスト・速度の面でも純増ではない。スクリプト型はスクリプト実行と構造化のためツール手数とレイテンシが増えるが（C3 tool 16.7 / lat 65.8 秒）、生成（output）トークンはスクリプト出力のパススルー＋構造化 JSON にとどまり、判定文を長々と生成する推論型より制御しやすい。決定的に効くのはモデル単価で、スクリプト型は弱モデルでも品質が落ちないため、「推論型×強モデル（C2 $0.2635/rep）」と同等以上の実質正答を「スクリプト型×弱モデル（C3 $0.0641/rep）」で達成でき、品質を保ったままモデル単価を大幅に下げられる。静的評価フィードバック 1 回が、精度・安定性に加えて運用コスト構造の改善まで連鎖した。

### 6.4 RQ4: 思考予算レバー vs 構造的解

推論型の確率的な脆弱性は、判定方式を変えずに思考予算を上げるだけでも救えた。C2H は acc 3.933 → 4.000（d = 0.26）・全ケース distinct=1 に達し、expense-sum の算術も誤り 1 → 0 に締まった。しかしこのレバーには 2 つの限界がある。第一に**コスト**: C2H は 5 セル中で最も高い（$0.2985/rep、レイテンシ平均 99.7 秒）。同じ満点・同じ決定性を、スクリプト型は最も弱く最も安いモデル（C3 $0.0641/rep）で達成する。第二に**保証の質**: 思考予算で得た決定性は強モデルが追加の計算で都度たどり着いた確率的な収束であって、弱モデルには効かず（思考予算は C1 相当の弱モデルの崩壊を埋めない）、入力規模やモデル更新に対する構造的な保証ではない。対してスクリプト型の決定性は判定が版管理された決定的スクリプトである以上モデル能力に依存しない。**「計算資源で殴る」は強モデル前提の応急策、「判定を正規の決定的スクリプトに委ねる」はモデル非依存の構造的解**であり、両者は満点という結果は同じでもコストと頑健性で質的に異なる。

### 6.5 RQ3: なぜ静的評価か — 予測妥当性とコスト非対称性

monosashi の静的スコアは推論型 1.65 → スクリプト型 3.05（/4）と改善し、その指摘事項（再現性保証・テスト・境界検証・ルール正本の欠如）は、実行評価で推論型が実際に崩れた箇所（決定性・算術・ルール適用）と対応していた。静的評価の重要性は次の 3 点に整理できる。

1. **実行前にリスクを特定する**: 本実行評価は正解データ（`ref/`）・グレーダ・実行ハーネス・計 150 rep（参考換算 $28.69）を要した。静的評価は成果物を読むだけでこのインフラを一切要さず、同じ弱点をスクリプト型実装の**前に**指摘した。
2. **実行評価が観測しにくいリスクを捉える**: 強モデルでの実行評価だけでは推論型の弱点はほぼ見えない（C2 acc 3.933/4）。「モデルを弱くしたら崩れる」という条件依存の脆弱性は、静的評価が設計構造（判定の所在・正本・テストの有無）から指摘できる種類のリスクである。
3. **改善の方向を与える**: 静的評価の指摘はスコアだけでなく改善項目（外部化・正本化・テスト・境界検証）として実装可能な形で返り、スクリプト型はそれをそのまま反映して全指標の改善に至った。

一方で静的評価は効果量を予言しない（スコア 1.65→3.05 は d = 5.25 という実行時の差の大きさを直接は意味しない）。構造的リスクの特定は静的評価、効果の定量化は実行評価、という補完関係で両レイヤーを併用するのが有効である。

## 7. 限界と妥当性への脅威

1. **gold の出自バイアス**: gold はスクリプト型の `audit.mjs` 出力であり、accuracy 軸は構造上スクリプト型有利になりうる。本評価はスクリプト型実装固有の出力規約（reason/error 列挙の順序・併記網羅）を採点から除外した acc を用いることでこの歪みを回避したが、判定基準の定義自体がスクリプト型実装と同一である点は残る。なお expense-sum の正解値 98,810 は実装に依存しない客観的な算術値であり、推論型の算術崩壊の測定は公平である。
2. **フィードバック効果の分離**: 推論型→スクリプト型の差分は静的評価の指摘に基づくが、改善の実装には開発者（Opus 4.8）の裁量が含まれ、「静的評価フィードバック固有の効果」と「再実装一般の効果」を完全には分離できない（フィードバックなしの再実装という対照群は未実施）。
3. **ケースの代表性**: 難問 4 ケース・単一ドメイン（経費・請求書）での結果であり、一般化には他ドメインでの追試を要する。基本難度ケースでは推論型も通過するため、本結果は「難条件での挙動差」に関する主張である。
4. **思考予算の単一水準**: C2H の思考予算は high（16,000 tokens）の 1 水準・opus のみで、予算量との用量反応や弱モデルでの効果は測っていない。「思考予算では弱モデルの崩壊を埋められない」は C1 への未適用からの推論であり直接の実測ではない。
5. **persona 再現とモデル・時点の限定**: 各 rep は配備版エージェントそのものではなくその挙動を忠実に再現した persona プロンプトのサブエージェント実行であり、モデルは 2 種（Haiku 4.5 / Opus 4.8）、実行は同一期間に集中している。プロバイダ側の変動（モデル更新等）は統制外。
6. **USD は参考値**: 公開参考単価による換算であり、実運用（定額プラン）の実請求とは異なる。
7. **非構造化入力の清浄度**: テキストは構造化版と同一の値を一意に復元できる「きれいな自由テキスト」であり、OCR ノイズ・欠損・矛盾する重複記載を含まず、規模も最大 28 明細にとどまる。全セルが転記・正規化で崩れなかったのはこの難易度で抽出が飽和していることを示すものであり、より敵対的な入力（ノイズ入り OCR、数百明細、期間境界をまたぐ年省略日付等）での破綻点は本実験の範囲外である。

## 8. 結論

経費精算エージェントを対象に、判定を LLM の推論（と随時の自前スクリプト）で行う**推論型**と静的評価フィードバックに基づき判定を版管理された決定的スクリプトに外部化した**スクリプト型**を、モデル能力 2 水準・全セル非構造化テキスト入力・各セル N=30 で実行評価し（C1〜C4）、さらに推論型/opus の思考予算を上げた追加セル（C2H）を加えて、静的評価フィードバックの効果と「計算資源で殴る」レバーとの比較を行った。スクリプト型は、(1) 精度: 弱モデルで acc 2.067/4 → 4.000/4 と崩壊（多数整数の合算 8/30・誤値 16 種を含む）を解消し、(2) 安定性: 両モデルで全ケース distinct=1 の完全な決定性を獲得し、(3) コスト: 満点群で最安（$0.0641/rep）に到達した。推論型の脆弱性は思考予算を上げても救えるが（C2H: acc 3.933 → 4.000・全ケース distinct=1、d = 0.26）、それは全セル中最高コスト（$0.2985/rep）の強モデル前提の応急策であって、弱モデルにも効くモデル非依存の構造的保証はスクリプト型側にしかない。静的スコアの改善（1.65 → 3.05）は実行時の改善箇所を実行前に正しく予告しており、強モデルでの実行評価では観測しにくい「モデル能力依存」という構造的リスクを実行インフラなしで特定できる点に静的評価の固有の価値がある。構造的リスクの特定を静的評価が、効果の定量化を実行評価が担う 2 レイヤーの品質保証、および「LLM は構造化に、コードは判定に」という分業設計が、業務向け LLM エージェント開発の実践的な型として支持される。
